Section 1: ADA/WCAG 2.1 AA Compliance Checklist

As of April 2026, the ADA Title II rule requires WCAG 2.1 Level AA compliance for all public-facing websites. This section provides a comprehensive checklist to ensure your website meets accessibility standards.

1.1 Perceivable

Content must be perceivable to all users, regardless of sensory abilities.

Images & Alt Text

  • Every image has descriptive alt text (150 characters or less)
  • Alt text describes what the image shows, not just "image" or "photo"
  • Decorative images have empty alt attributes (alt="")
  • Complex images (diagrams, charts) have longer descriptions or adjacent text
  • Screenshots of forms include alt text describing the content
Good Alt Text Examples:
❌ Bad: "Dr. Patel"
✓ Good: "Dr. Neil J. Patel, MD, MBA, board-certified neurologist specializing in sports neurology"

❌ Bad: "Team"
✓ Good: "University athletics team meeting with Dr. Patel for concussion consultation"

Video & Audio Content

  • All videos have captions (burned-in or closed captions enabled)
  • Captions are synchronized with audio
  • Transcripts provided for video content (published or downloadable)
  • Audio descriptions provided for important visual content in videos
  • Podcasts or audio lectures have published transcripts

Color & Contrast

  • Normal text (under 18px): 4.5:1 contrast ratio with background
  • Large text (18px+ bold or 24px+ regular): 3:1 contrast ratio
  • Color is never the only way to convey information (use text labels, patterns, or icons)
  • Links are distinguishable from surrounding text (not color alone)
  • Focus indicators are visible and have sufficient contrast
Your Dark Theme Compliance:
• Navy #0a1628 + White text: 19.4:1 ✓ (exceeds 4.5:1)
• Navy #0a1628 + Blue #2d7ff9: 7.8:1 ✓ (exceeds 4.5:1)
• All color combinations meet WCAG AA standards

Text Resizing

  • Page remains readable and functional at 200% zoom
  • No horizontal scrolling required at 200% zoom (except large tables)
  • Squarespace responsive design automatically accommodates text resizing
  • Test in browser: use Ctrl/Cmd + Plus to zoom to 200%

1.2 Operable

All functionality must be operable via keyboard and other input methods.

Keyboard Navigation

  • All interactive elements (buttons, links, forms) are keyboard accessible
  • Tab order is logical and follows visual reading order
  • Focus indicator is clearly visible on all focusable elements
  • No keyboard traps (elements where focus can't be moved away)
  • Form inputs accept keyboard input without mouse requirements

Skip Navigation

  • Skip to Main Content link is the first focusable element on page
  • Skip link is visible on focus (even if hidden by default)
  • Skip link actually skips repetitive navigation content
  • Skip links target the main content area with id="main"

Motion & Animation

  • No content flashes more than 3 times per second (flash rate < 3Hz)
  • Animated GIFs are static or have animations < 5 seconds
  • Auto-playing videos/audio have pause/stop controls
  • Animations are essential, not purely decorative
  • Parallax effects or motion animations include prefers-reduced-motion support

Timing & Timeouts

  • Session timeouts have warnings and can be extended
  • Forms don't auto-submit after user inactivity
  • Countdown timers (e.g., for booking) provide at least 20 seconds
  • Users have enough time to read and interact with content

Page Titles & Navigation

  • Every page has a unique, descriptive page title in <title> tag
  • Page title appears in browser tab and matches page purpose
  • Multiple ways to find pages: navigation menu, search, sitemap
  • Breadcrumbs help users understand site structure
  • Navigation menu is consistent across all pages
Page Title Examples:
• Homepage: "Dr. Neil J. Patel | Sports Neurologist & Concussion Specialist"
• About: "About Dr. Neil Patel | Board-Certified Sports Neurologist"
• Concussion Info: "Concussion Treatment & Return-to-Play Protocol | Dr. Patel"
• Contact: "Contact Dr. Neil Patel | Sports Neurology Consulting"

1.3 Understandable

Content must be understandable to all users.

Language & Structure

  • HTML tag includes language attribute: <html lang="en">
  • Changes in language within content are marked up (e.g., <span lang="es">)
  • Heading hierarchy is logical (H1 → H2 → H3, no skipped levels)
  • Each page has exactly one H1 tag
  • Headings describe page sections, not formatting

Navigation Consistency

  • Navigation appears in same location on every page
  • Navigation menu items are in same order across pages
  • Navigation link labels are consistent (don't rename pages)
  • Site search works the same way throughout site

Forms & Error Handling

  • All form inputs have associated <label> elements
  • Label text is visible and clearly associated with input field
  • Form instructions are clear before submission
  • Error messages identify the specific field with an error
  • Error messages suggest how to fix the error
  • Form fields with required information are marked as required
  • Form validation allows correcting errors without resubmitting entire form
Form Label Example:
<label for="email">Email Address:</label>
<input type="email" id="email" name="email" required>

✓ Correct: Label is associated with input via for attribute
❌ Wrong: Placeholder text as only label; no <label> element

Predictable Behavior

  • Navigation doesn't cause unexpected context changes
  • Submitting forms doesn't navigate to unexpected page
  • Opening links doesn't open new tabs unexpectedly
  • Auto-submitting forms requires explicit user action

1.4 Robust

Content must be robust and compatible with assistive technologies.

HTML Validity

  • HTML validates with no errors using W3C Validator
  • All tags are properly closed
  • IDs are unique on each page
  • Proper semantic HTML: <nav>, <main>, <article>, <section>

ARIA & Landmarks

  • ARIA landmarks (role="navigation", role="main", role="contentinfo") are used appropriately
  • ARIA attributes have valid values
  • aria-label is used only when visible text isn't available
  • aria-describedby links inputs to descriptions when needed
  • ARIA doesn't contradict semantic HTML

Screen Reader Compatibility

  • All content is announced properly by screen readers (VoiceOver, NVDA, JAWS)
  • All UI components have name, role, and value (accessible properties)
  • Hidden decorative content is marked with aria-hidden="true"
  • Dynamic content updates are announced via aria-live regions
  • Icon buttons have accessible labels (not just the icon)
Icon Button Example:
❌ Wrong: <button><i class="icon-search"></i></button>
✓ Good: <button aria-label="Search"><i class="icon-search" aria-hidden="true"></i></button>

1.5 Squarespace-Specific ADA Implementation

Built-in Features

  • Squarespace templates are WCAG AA compliant by default
  • Responsive design works automatically for keyboard and mobile
  • Form elements built into Squarespace include labels automatically
  • Images and video blocks have alt text fields
  • Navigation is keyboard accessible by default

What You Must Do

Critical Squarespace ADA Tasks:

1. Add Alt Text to All Images
For every image on your site, fill in the alt text field in Squarespace. This is the #1 accessibility issue on most websites.

2. Use Proper Heading Hierarchy
In text blocks, use Squarespace's heading styles (H1, H2, H3) instead of making text large with formatting. Never skip heading levels (e.g., H1 → H3, skipping H2).

3. Add an Accessibility Statement
Create a new page titled "Accessibility" that states your commitment to accessibility. Example: "Dr. Patel is committed to making this website accessible to everyone. If you encounter any accessibility issues, please contact us."

4. Use Form Elements Properly
Use Squarespace's built-in contact forms and form blocks. Always assign labels to inputs. Don't use text blocks as fake labels.

5. Test with Keyboard Navigation
Use only the Tab key to navigate your entire website. Verify all buttons, links, and forms work. Check that focus indicators are visible.

6. Test on Mobile
Use Squarespace's mobile preview to test touch targets, readability, and color contrast on small screens.

Accessibility Overlay Tools (Optional Supplement)

Some sites use third-party accessibility overlays (e.g., UserWay, accessiBe) that add extra controls for users. Important: These are supplements, not replacements. Your site must be accessible first; overlays provide additional convenience.

  • Never rely on an overlay as your only accessibility solution
  • If using an overlay, it should supplement, not hide, native accessibility
  • Overlays may add text sizing, color filters, and reading guides
  • Consider whether an overlay is worth the performance impact and complexity
  • Some overlays may conflict with browser built-in accessibility features

Section 2: SEO Strategy for Sports Neurologist

SEO for medical professionals requires a balance between search visibility and regulatory compliance. Focus on providing valuable, accurate information that attracts patients and referral sources.

2.1 Technical SEO Foundation

Site Speed & Performance

  • Page load time < 3 seconds on 4G mobile (test with Google PageSpeed Insights)
  • Optimize images: compress, use modern formats (WebP), lazy load below fold
  • Use Squarespace's built-in CDN (included automatically)
  • Minimize CSS and JavaScript (Squarespace does this automatically)
  • Enable browser caching (Squarespace handles this)
  • Remove unused fonts (use system fonts or limit to 1-2 web fonts)

Mobile-First Responsive Design

  • Site is fully responsive and works perfectly on all screen sizes
  • Touch targets (buttons, links) are at least 48x48 pixels
  • Viewport meta tag is set: <meta name="viewport" content="width=device-width, initial-scale=1.0">
  • Text is readable without zooming on mobile (minimum 16px)
  • Squarespace templates are mobile-first by default

SSL Certificate & HTTPS

  • All pages served over HTTPS (required for Google ranking, medical compliance)
  • SSL certificate is valid and not expired
  • No mixed content (all resources loaded over HTTPS)
  • Squarespace provides free SSL certificates

XML Sitemap & Robots.txt

  • XML sitemap exists at /sitemap.xml
  • Sitemap includes all important pages (exclude admin, duplicate, or thin content)
  • Sitemap is submitted to Google Search Console
  • Robots.txt file at root: /robots.txt
  • Robots.txt allows search engines to crawl public content
  • Squarespace generates sitemap automatically

Structured Data & Schema Markup

  • Add Schema.org JSON-LD for key page types
  • Use Physician schema for about page
  • Use MedicalSpecialty schema for service pages
  • Use FAQPage schema for FAQ sections
  • Use LocalBusiness schema if location-specific
  • Test schema markup with Google's Rich Results Test
Physician Schema Example:
<script type="application/ld+json">
{
  "@context": "https://schema.org/",
  "@type": "Physician",
  "name": "Dr. Neil J. Patel",
  "description": "Board-certified Sports Neurologist specializing in concussion management",
  "medicalSpecialty": "Sports Neurology",
  "credentials": "MD, MBA, UCNS Board Certified",
  "url": "https://www.drneilpatelmd.com",
  "email": "contact@drneilpatelmd.com"
}
</script>

Canonical URLs & URL Structure

  • Each page has a canonical URL (self-referential or pointing to preferred version)
  • URLs are clean, descriptive, and lowercase: /concussion-treatment not /page123
  • URLs don't change (or implement 301 redirects if they do)
  • No duplicate content across multiple URLs
  • Squarespace generates clean URLs automatically

Heading Hierarchy & Page Structure

  • One H1 per page (matches page title)
  • H1 describes main topic of page
  • H2 and H3 headings follow logical hierarchy (no skipped levels)
  • Headings include target keywords naturally (not forced)
  • Headings are descriptive, not clickbait-y

Internal Linking Strategy

  • Use descriptive anchor text (keywords where relevant, not "click here")
  • Link related pages together (e.g., Concussion page links to Return-to-Play page)
  • 1-3 internal links per 500 words of content
  • Use breadcrumbs for site navigation
  • Hub pages (e.g., Services) link to pillar content (Concussion, Brain Injury)

Image Optimization

  • Image filenames are descriptive: dr-neil-patel-sports-neurologist.jpg (not image1.jpg)
  • Image alt text includes keywords naturally (not stuffed)
  • Image file sizes are optimized (< 200KB for web)
  • Use modern image formats: WebP with PNG/JPG fallback
  • Include image captions where relevant

3.7 Performance & Security Best Practices

Image Optimization in Squarespace

  • Squarespace auto-optimizes images for web
  • Use source images that are not larger than 2x their display size
  • Compress images before uploading (tools: TinyPNG, ImageOptim)
  • Avoid uploading photos larger than 2-3 MB
  • Use descriptive image file names for SEO
  • Always add alt text to images

Security & Compliance

  • SSL Certificate enabled (free with Squarespace, automatic HTTPS)
  • Keep Squarespace CMS updated (automatic)
  • Use strong password for Squarespace account (20+ characters, mixed case, numbers, symbols)
  • Enable two-factor authentication on Squarespace account
  • Privacy policy is clear (required by GDPR, CCPA)
  • Terms of service updated (if accepting patients/clients online)
  • No personal health information in contact forms (use HIPAA-compliant channels)
